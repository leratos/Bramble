# Bramble – Journal

Solange Bramble selbst noch nicht läuft, wird dieses Journal klassisch
als Textdatei gepflegt. Ab Phase 4 (DB läuft, Connector eingerichtet)
wandert Brambles eigenes Journal in die zentrale Datenbank.

Format: chronologisch, neueste Einträge unten. Statusangaben:
in_arbeit | abgeschlossen | notiz | bugfix.

Bewusst NICHT in Phase 1 enthalten (Folgephasen):
- MCP-Server (Phase 2): FastMCP-Bindings, Tool-Definitionen
- Async-Variante (Phase 2 optional): Tausch sqlite3 -> aiosqlite
- Auth, Rate-Limit, Logging, Deployment (Phase 3)

Bekannte technische Schulden / offene Punkte:
- list_projects() liefert nur Namen, keine Eintrags-Counts oder
  letzten Timestamp. Für die geplante MCP-Tool-Variante in Phase 2
  müssen wir entscheiden, ob das ausreicht oder mehr Metadaten gebraucht
  werden.
- search() verwendet rohe FTS5-Syntax. Wenn Bramble von außen
  (Claude.ai-Connector, Phase 4) erreichbar ist, sollten wir prüfen,
  ob wir Queries mit Anführungszeichen vorab quoten, um Power-User-
  Operatoren bewusst zu erlauben/zu verbieten.
- Keine WAL-Mode-Aktivierung. Solange Bramble nur ein einzelner
  Prozess ist, irrelevant. Phase 3 (parallele Requests gegen den
  systemd-Service) müssen wir das nochmal evaluieren.
- Es gibt keine Migrations-Versionierung (alembic/yoyo). Schema ist
  per CREATE TABLE IF NOT EXISTS idempotent, aber Spalten-Änderungen
  später erfordern ein eigenes Migrations-Konzept.

Branch ist commitet, aber NICHT gepusht (Repo auf GitHub legt
der Nutzer selbst an, dann manuell remote add + push).
